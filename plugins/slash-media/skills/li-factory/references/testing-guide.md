# Skill 测试与评估指南

当需要严格测试 skill 效果时参考本文档。

---

## 测试用例格式

保存到 `evals/evals.json`：

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户会说的真实请求（要具体、有背景、有细节，不要抽象）",
      "expected_output": "预期结果描述",
      "assertions": []
    }
  ]
}
```

**好的测试 prompt 特征**：
- 有具体背景（文件名、公司名、使用场景）
- 长度适中，接近真实用户请求
- 包含边缘情况和歧义场景

**坏的测试 prompt**：
- `"分析这个数据"` — 太抽象
- `"创建一个 skill"` — 太短，不像真实请求

---

## 断言（Assertions）

对有客观输出的 skill 设计断言（文件生成、格式正确、数据提取等）。对主观输出（写作风格、创意质量）不要强制断言，改用人工评估。

好的断言要求：
- 名称描述性强，一眼就懂在检查什么
- 客观可验证，不依赖主观判断

```json
{
  "assertions": [
    {
      "text": "输出文件包含 YAML frontmatter",
      "passed": true,
      "evidence": "文件第1行是 ---"
    }
  ]
}
```

---

## Description 优化

当 skill 触发不准确时，生成触发测试集：

**应触发（8-10 条）**：
- 不同表达方式覆盖同一意图
- 包含不直接命名 skill 但明确需要它的场景
- 包含口语化、有错别字、大小写混乱的写法

**不应触发（8-10 条）**：
- 最有价值的是"近似场景"——共享关键词但需要其他工具
- 不要放明显不相关的测试（没有测试价值）

**测试集示例格式**：
```json
[
  {"query": "帮我看看这个选题能不能火", "should_trigger": false},
  {"query": "我想做一个自动生成周报的 skill", "should_trigger": true}
]
```

---

## 迭代改进原则

**从反馈中泛化，而非过拟合**：skill 要服务无数次调用，不要只修复测试用例里暴露的问题，要理解背后的模式。

**读 transcript，不只看输出**：如果 Claude 在每次测试中都独立写了同一个辅助脚本，这是信号——把脚本放进 `scripts/`。

**解释原因而非强制命令**：与其写 `MUST always do X`，不如解释"因为 Y，所以做 X"。LLM 理解推理比死记规则更有效。

**保持精简**：删除不起作用的内容。如果某段指令让 Claude 做无用功，删掉它。
